> Recenzia spec-u `docs/specs/m2.5-anchor-format.md` (branch `m2.5-anchor-spike`).
> Zistenia a odporúčania zoradené podľa závažnosti.
> Dátum: 2026-06-21

---

## Celkové hodnotenie

Spec je vecne dobrý a trafil najdôležitejšie rozhodnutia:

- **Investigácia pred dizajnom** (sekcia 3 — spike, ktorý zistí, čo `EpubController`
  reálne vracia) je správny prvý krok. „Bez tohto kroku je každý ďalší dizajn špekulácia"
  je presný inštinkt.
- **Normalizácia** je premyslená — NFC áno, ale **lowercase nie a diakritika sa nestrip**
  (š/č/ž, talianske akcenty). Toto väčšina implementácií pokazí.
- **Doménová vrstva** bez Flutter/Drift závislostí, **testy nad reálnymi fixtures** vrátane
  diakritiky, spike sa po dokončení maže. Dobrá disciplína (DoD, milestone tabuľka).

Nižšie sú diery, ktoré stoja za opravu — zoradené podľa dopadu.

## P1 — Vecné problémy (vyriešiť pred S2)

### P1.1 `ReadingPositionAnchor` NIE JE render-agnostický

**Problém:** `progress` (0.0–1.0) sa resolve-uje cez `progress * chapter_render_height`.
Render height ale závisí od veľkosti písma, margins, line-height a šírky viewportu.

- Telefón, veľký font → kapitola 8000px → `0.42` = odsek X
- PC, malý font → kapitola 3000px → `0.42` = úplne iný odsek
- Aj len zmena fontu na tom istom zariadení to posunie

Tým spec porušuje vlastný cieľ (reflow-nezávislosť) a deje sa to pri **každom** sync
pozície, takže ten papercut je stály. Pri pozícii to nie je fatálne ako pri highlighte,
ale je zbytočné.

**Fix (lacný):** `progress` rátaj ako zlomok **dĺžky normalizovaného textu kapitoly**,
nie render height. Znak 5000 z 12000 = 0.42 nezávisle od toho, ako sa to kreslí.

```dart
class ReadingPositionAnchor {
  final String chapterId;   // manifest href; fallback "#<spineIndex>"
  final int charOffset;     // offset v NORMALIZED texte kapitoly
  final int chapterLength;  // dĺžka NORMALIZED textu (pre výpočet % a UI)
  // progress = charOffset / chapterLength  → render-agnostické
}
```

JSON: `{"c": "...", "o": 5000, "n": 12000}`

**Ešte lepšie:** ak spike ukáže, že `epub_view` vie povedať najvrchnejší viditeľný text,
pozícia môže byť degenerovaný text-anchor (quote prvej vety viditeľného odseku) a
resolve-ne sa presne ako highlight.

### P1.2 Offset vs. normalizovaný text — skrytý bug v `createHighlight`

**Problém:** metóda berie `selectionStart/End` ako offsety v *normalizovanom* texte, ale
selekcia z rendereru je v súradniciach *pôvodného* textu. Normalizácia mení dĺžku reťazca
(collapse whitespace, strip soft-hyphens), takže offsety nesedia →
`normalize(chapterText).substring(selectionStart, selectionEnd)` vráti zlý výrez.

**Fix (čistejší — string-first):** ak spike ukáže, že `onSelectionChanged` vracia samotný
**vybraný string** (nie len offsety), postav anchor čisto zo stringov a offsety pri tvorbe
nepotrebuj vôbec:

```dart
static HighlightAnchor createHighlight({
  required String chapterId,
  required String rawChapterText,
  required String rawSelectedText,
  required String rawPrefix,   // okolitý text z rendereru
  required String rawSuffix,
}) {
  final quote  = normalize(rawSelectedText);
  final prefix = normalize(rawPrefix);
  final suffix = normalize(rawSuffix);
  final normalizedChapter = normalize(rawChapterText);
  // charOffset je len HINT — dorátaný hľadaním, nie nárok
  final hint = normalizedChapter.indexOf(prefix + quote + suffix);
  return HighlightAnchor(
    chapterId: chapterId,
    quote: quote,
    prefix: prefix.length > 32 ? prefix.substring(prefix.length - 32) : prefix,
    suffix: suffix.length > 32 ? suffix.substring(0, 32) : suffix,
    charOffset: hint >= 0 ? hint + prefix.length : null,
  );
}
```

**Akcia:** do investigačnej tabuľky (sekcia 3.3) pridať riadok
„selection vracia samotný text, alebo len offsety?". Podľa odpovede sa rozhodne,
či treba mapping krok, alebo stačí string-first.

## P2 — Doplnky do ADR 0002 (jedna veta každý, ušetria prepis vo Fáze 5)

### P2.1 Stabilita `chapterId` cez href je oversold

Href je stabilný cez kópie **toho istého súboru** a re-import, ale **nie** cez rôzne edície
/ zdroje toho istého titulu (iné cesty, iné delenie kapitol, iný preklad).

**Akcia:** explicitne napísať a previazať s book-identity — *anchor sa resolve-ne v rámci
súboru s rovnakým content hashom; cross-edícia nie je podporovaná.* Inak budeš od anchoru
čakať niečo, čo nikdy nedodá.

### P2.2 Merge `max(progress)` má dieru v ordering-u

Progress je per-kapitola, pravidlo neporovnáva kapitoly. (A: kap. 5 @ 0.1, B: kap. 3 @ 0.9
→ `max` vyberie B, hoci A je ďalej.) Na „kto je ďalej" treba **spine index kapitoly** ako
primárny kľúč ordering-u, až potom progress.

- Href ostáva identita kapitoly; spine poradie dorátaj z aktuálneho manifestu
  (poradie v spine je vlastnosť súboru, nie anchoru).
- „max vyhráva" je produktové rozhodnutie („najďalej prečítané" vs. „posledná skutočná
  pozícia"), nie samozrejmosť — pomenuj ho v ADR.

### P2.3 Identita anotácie chýba (rozhodnúť teraz, ukladá sa vo Fáze 3)

`HighlightAnchor` je správne čistý lokátor bez id. Ale *záznam anotácie* musí mať
klientom generované stabilné **UUID + created/updated timestamp + device id** — inak vo
Fáze 5 nededupe-uješ a nezmerguješ highlighty z dvoch zariadení, ani nenaviažeš úpravy/
zmazania ako eventy.

**Akcia:** jedna veta v sekcii dôsledkov ADR — oddelenie „anchor = kde, annotation record
= čo + identita", nech Fáza 3 schéma vznikne s týmito poľami od začiatku.

## P3 — Vedome odložiteľné (ak sa to v ADR prizná)

### P3.1 Selekcia nad 200 znakov

Cap na 200 znakov bez pravidla, čo sa stane s dlhšou selekciou — pri resolve stratíš koniec
rozsahu. Robustnejšie: **ukotviť oba konce** (start-quote + end-quote, à la W3C
`RangeSelector`) a zvýrazniť span medzi nimi. Pre v1 sa dá puntnúť, ak sa to prizná.

### P3.2 Fuzzy threshold — pripni metriku

„Levenshtein ratio ≥ 0.92" — pripni presne, ktorá metrika (normalizovaný Levenshtein vs.
difflib `SequenceMatcher.ratio()` dávajú rôzne čísla pri tom istom prahu) a pridaj
tie-break pravidlo (najlepší ratio; pri zhode najbližšie k `charOffset`), nech sú testy
deterministické.
