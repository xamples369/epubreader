## 1. Cieľ a motivácia

Pred Fázou 3 (anotácie) a Fázou 5 (cloud sync) potrebujeme vyriešiť otázku:

> **Ako reprezentujeme bod / rozsah v knihe tak, aby zostal stabilný naprieč
> zariadeniami, zmenami fontu / margins, a prípadnou zmenou EPUB súboru?**

Bez stabilnej, **reflow-nezávislej** kotvy nemá zmysel stavať highlighty ani sync —
po prvej zmene veľkosti písma alebo prenose knihy na druhé zariadenie sa všetky
pozície rozsypú a hodnota produktu sa stratí.

V ADR 0001 (M1) sme zvolili `epub_view` ako renderer kvôli stabilite a kvalite
vykresľovania, ale **nehodnotili sme aký lokátor knižnica vie poskytnúť**. Tento
spec zatvára tú medzeru.

Strategy doc `docs/strategy/2026-06-21-renderer-anchor-sync-analysis.md` (sekcia 2)
identifikoval túto medzeru ako blokujúce riziko pre celý sync. M2.5 je odpoveď.

## 4. Anchor formáty

### 4.1 `ReadingPositionAnchor`

```dart
class ReadingPositionAnchor {
  final String chapterId;     // primárny: manifest href; fallback: "#<spineIndex>"
  final int charOffset;       // offset v NORMALIZED texte kapitoly
  final int chapterLength;    // dĺžka NORMALIZED textu (pre UI % výpočet)
}
```

**JSON:**
```json
{"c": "OEBPS/Text/chapter01.xhtml", "o": 5000, "n": 12000}
```

**Render-agnostické:** `progress = charOffset / chapterLength` sa ráta zo *znakovej*
dĺžky normalizovaného textu, nie z rendered výšky kapitoly. To znamená že
„znak 5000 z 12000" znamená to isté na PC s malým fontom aj na mobile s veľkým —
nezávisle od reflow.

> **Pozn.:** pôvodný návrh používal `progress 0.0–1.0` mapovaný na `chapter_render_height`.
> To bolo *render-závislé* (telefón vs PC dáva inú render height pre tú istú kapitolu)
> a porušovalo cieľ reflow-nezávislosti. Char-offset variant tento problém rieši.

**Prečo `chapterLength` ukladáme v anchore aj keď je vlastnosť súboru:** purely
pre **offline UI % display** v zoznamoch kníh / sync conflict resolution kde
nemáme k dispozícii aktuálnu knihu (napr. cloud sync log entry sa zobrazuje
v Settings histórii pred otvorením knihy). Pri *resolve* sa `chapterLength`
v anchore IGNORUJE — vyberie sa aktuálna dĺžka z aktuálneho súboru, lebo ten je
pravdivý. Anchor `chapterLength` je tým pádom najlepší effort cache, môže byť
stale (ak používateľ pridal kapitolu re-importom), ale to nevadí — re-rátanie
pri resolve to opraví. Bez toho by sme pri zobrazovaní pokroku v UI museli vždy
otvoriť celú knihu, čo je drahé pre dlhú knižnicu.

**Resolve:**
1. Nájdi `chapterId` v manifest-e otvoreného EPUB-u.
2. Prepni na kapitolu, načítaj kanonický text (§4.5) a získaj aktuálne `chapterLength`.
3. **Mapni `charOffset` na pozíciu v render-e — make-or-break neznáma.** Záleží od
   výsledku spike-u (§3.3):
   - **Plán A (preferovaný):** `EpubView` controller vie programaticky scrollovať
     priamo na char N v texte kapitoly → priame char→scroll mapovanie.
   - **Plán B (fallback ak A nie je dostupný):** ReadingPositionAnchor sa stáva
     **„degenerovaným HighlightAnchor"** — pri tvorbe vezme prvú vetu okolo
     `charOffset` ako quote a uloží sa cez HighlightAnchor mechanizmus. Resolve
     potom volá `findHighlight` a scrolluje na widget ktorý obsahuje nájdený text.
     Mierne pomalšie ale stále funguje.
   - **Rozhodnutie medzi A a B sa zafixuje v ADR 0002** po dokončení spike-u.
     Plán B je explicitne v scope — ak A nie je dostupné, nevedieme to ako blocker
     ale prepneme.
4. Scroll/prejdi na ňu.

**Voliteľný upgrade (závislý od spike v sekcii 3):** ak `epub_view` vie poskytnúť
najvrchnejší viditeľný text, môžeme `ReadingPositionAnchor` neskôr degradovať na
„degenerovaný" HighlightAnchor (quote prvej viditeľnej vety) a resolve cez rovnaký
text-based algoritmus. Toto rozhodnutie ide do ADR 0002 po spike-u.

**Sync merge — produktové rozhodnutie:** voľba medzi:
- **„Furthest read wins"** — `(spine_index_of_chapter, charOffset)` lexikograficky,
  posledná = najďalej prečítaná. Vhodné keď nás zaujíma „kam som dočítal".
- **„Last wins"** — vyberie sa event s najnovším timestamp-om, bez ohľadu na pozíciu.
  Vhodné keď používateľ skáče dozadu (re-read kapitoly).

**Default pre MVP:** *furthest read wins* — to je čo používateľ čaká pri synchronizácii
medzi PC a mobilom. „Last wins" je odložené pre Fázu 5 ako voliteľnosť v Settings.

**Ordering naprieč kapitolami:** porovnanie vyžaduje **spine index** (poradie kapitoly
v manifest-e), nie len `chapterId`. Spine index sa získa z aktuálneho EPUB-u pri
resolve, nie sa ukladá v anchore (to je vlastnosť súboru, nie anchoru).

### 4.2 `HighlightAnchor`

```dart
class HighlightAnchor {
  final String chapterId;     // rovnaký formát ako vyššie
  final String quote;          // exact text snippet, normalised, max ~200 znakov
  final String prefix;         // 32 znakov pred quote (context)
  final String suffix;         // 32 znakov za quote (context)
  final int? charOffset;       // hint: očakávaná pozícia v kapitole (po normalizácii)
}
```

**JSON:**
```json
{
  "c": "OEBPS/Text/chapter01.xhtml",
  "q": "Down, down, down. Would the fall never come to an end?",
  "pre": "There was nothing else to do, so Alice soon began ",
  "suf": " 'I wonder how many miles I've fallen by this time?'",
  "o": 1247
}
```

**Limity:**
- `quote` max 200 znakov. **Dlhšie selekcie:** v MVP cap-neme na 200 znakov
  (zoberieme prvých 200) a zalogujeme warning. Cena: stratený koniec rozsahu pri
  veľmi dlhej selekcii. Robustnejšia varianta — ukotvenie oboch koncov ako
  start-quote + end-quote (W3C `RangeSelector` štýl) — je vedome **odložená do
  Fázy 3 alebo neskôr**; pridáme ju keď bude reálny use case (napr. citácia odseku
  na export).
- `prefix` + `suffix` po 32 znakov (dostačujúce na disambiguáciu vo väčšine prípadov)
- `charOffset` je *hint* nie nárok — anchor je validný aj bez neho

### 4.3 Voľba `chapterId`

**Primárne:** `manifest href` (napr. `OEBPS/Text/chapter01.xhtml`)
- Stabilné cez kópie **toho istého súboru** a re-import
- Spine index sa môže meniť keď vydavateľ pridá/odstráni front-matter
- Konzistentné s tým ako `epub_view` aj `epubx` knihy navigujú

**Fallback:** `spine index` ako stringifikované číslo s prefixom `#` (napr. `"#3"`)
- Použije sa keď manifest href nie je k dispozícii alebo sa nezhoduje

**Scope kde anchor platí:** anchor sa **resolve-uje v rámci EPUB súboru s rovnakým
content hashom** ako keď bol vytvorený. **Cross-edícia (iný preklad, iné delenie
kapitol, iný zdroj toho istého titulu) NIE JE podporovaná** — href v inom EPUB
súbore tej istej knihy môže byť úplne iný. Tento limit je akceptovaný a previazaný
na *book-identity* schému (`docs/specs/m2.6-book-identity.md`, ADR 0003 — vznikne
po M2.5). Anchor sa nesnaží riešiť to čo book-identity nevyriešila.

### 4.4 Normalizácia textu (`AnchorCodec.normalize`)

Pre konzistentné porovnávanie `quote`/`prefix`/`suffix` s textom kapitoly:

1. **Unicode NFC** (composed forms — `é` ako jeden codepoint, nie `e` + combining acute)
2. **Whitespace collapse** — všetky sekvencie whitespace (vrátane no-break-space,
   tabulátorov, newlines) → jediná medzera
3. **Trim** — odstránenie leading/trailing whitespace
4. **Strip soft hyphens** (U+00AD) — niektoré EPUB-y ich vkladajú pre hyphenation

Nie:
- Lowercase (case je sémanticky relevantný v citátoch)
- Diakritika strip (slovenské `š/č/ž` musia byť verne zachované)
- Punctuation strip (rozdiel medzi „Hello." a „Hello!")

### 4.5 Kanonický zdroj textu kapitoly (CRITICAL)

**Problém:** `createHighlight` (Fáza 3 — používateľ vyberie text v rendereri)
beží proti textu z **rendereru** (`epub_view` → `flutter_html` → naparsované HTML).
`findHighlight` (Fáza 3, 5 — resolve uloženého anchoru) beží proti textu z
**parsera** (`epubx`). Tieto dva môžu produkovať **rôzne normalizované reťazce**:

- `flutter_html` môže preskočiť alt text obrázkov, `figcaption`, footnote markery
- Poradie blokov a whitespace medzi elementmi sa môže líšiť (block-level vs inline)
- Render-text môže obsahovať pridané UI artefakty (page break markery, …)

Ak diverguje, anchor zachytený proti render-textu sa **systematicky nenájde**
v parser-texte. Fatálne pre celý systém — dokonca aj pri *exact* match na tej istej
knihe na tom istom zariadení.

**Rozhodnutie:** **jeden kanonický extraktor textu kapitoly** používaný **AJ
`createHighlight` AJ `findHighlight`**:

```dart

  /// Toto je JEDINÝ pravdivý zdroj textu pre anchor encoding/decoding.
  static String extract(EpubBook book, String chapterId) {
    // 1. Nájdi kapitolu podľa chapterId (href alebo #spineIndex)
    // 2. Vezmi HtmlContent
    // 3. Strip HTML tagy (zachovaj len text content)
    // 4. AnchorCodec.normalize()
    // 5. Vráť
  }
}
```

**Dôsledok pre API `createHighlight`:** signatúra už NEPRIJÍMA `rawChapterText`.
Volajúci (Fáza 3 UI) namiesto toho dodá knihu + chapterId, a `createHighlight`
si kanonický text vyžiada interne:

```dart
static HighlightAnchor createHighlight({
  required EpubBook book,
  required String chapterId,
  required String rawSelectedText,   // exact substring zo selekcie
  required String rawPrefix,         // okolitý text — môže byť z rendereru
  required String rawSuffix,         // okolitý text — môže byť z rendereru
});
```

Quote/prefix/suffix sa stále normalizujú (NFC + whitespace) — to nie je problém.
Problém bol len v tom proti čomu sa hľadá `charOffset` hint. Teraz to je
kanonický epubx text, takže `findHighlight` má proti čomu match-ovať to isté.

**Risk:** ak používateľ vyberie text ktorý existuje v rendereri ale nie v
epubx-derived texte (napr. footnote ktorý `epubx` nezahŕňa rovnako), anchor sa
vytvorí s `charOffset: null` a fuzzy match pri resolve nebude mať hint —
ale stále môže nájsť match. Worst case: anchor sa stratí. Akceptujeme, lebo
alternatíva (render-text ako kanonický zdroj) by zlomila resolve úplne.

**Spike závislosť (§3.3):** pridaný riadok testu „**vie sa selekcia z `epub_view`
nájsť v normalizovanom epubx texte kapitoly?**" — overí sa na všetkých 4 fixtures.
Ak na niektorých nie, vieme to TERAZ a buď nájdeme reconciliation pravidlá
(napr. strip footnote markery zhodne na oboch stranách), alebo prehodnotíme
extraktor. **Bez tohto testu ide M3 do rizika.**

## 5. `AnchorCodec` API

### Súborová organizácia

```
lib/domain/anchor/
├── reading_position_anchor.dart   (data class + toJson/fromJson)
├── highlight_anchor.dart          (data class + toJson/fromJson)
└── anchor_codec.dart              (createHighlight, findHighlight, normalize)
```

Anchor patrí do `lib/domain/` — čisté Dart triedy bez Flutter / Drift / DB
závislostí. Závisí len od `epubx` (parser knihy) a `dart:core`.

### Verejné API

```dart
class AnchorCodec {
  /// Vytvorí HighlightAnchor z RAW textu selekcie + RAW okolitého kontextu.
  /// String-first prístup — žiadne offset mapping medzi raw a normalised textom.
  /// Renderer (M3) volá `onSelectionChanged` ktoré vracia samotný vybraný text;
  /// rovnako vie poskytnúť pár desiatok znakov pred/za z DOM-u kapitoly.
  /// `book + chapterId` sa použijú na získanie KANONICKÉHO textu kapitoly (§4.5).
  static HighlightAnchor createHighlight({
    required EpubBook book,
    required String chapterId,
    required String rawSelectedText,
    required String rawPrefix,    // raw text PRED selekciou (môže byť ľubovoľná dĺžka)
    required String rawSuffix,    // raw text ZA selekciou (môže byť ľubovoľná dĺžka)
  });

  /// Resolve: nájdi anchor v aktuálnom obsahu knihy.
  /// Vracia AnchorRange v KANONICKOM normalised texte kapitoly, alebo null.
  static AnchorRange? findHighlight({
    required HighlightAnchor anchor,
    required EpubBook book,
  });

  /// Public helper — useful pre testy aj UI vrstvu.
  static String normalize(String text);
}

/// Vlastná typovo bezpečná trieda — neviažeme sa na Flutter `dart:ui.TextRange`
/// aby domain vrstva neimportovala Flutter.
@immutable
class AnchorRange {
  final int start;  // character offset v normalised texte
  final int end;
  const AnchorRange(this.start, this.end);
}
```

**Prečo string-first namiesto offsetov:** raw selekcia z rendereru je v offsetoch
*pôvodného* textu (alebo dokonca len ako samotný string). Normalizácia mení dĺžku
(collapse whitespace, strip soft hyphens), takže offsety v raw ≠ offsety v
normalized. String-first variant problém obíde úplne — pracujeme s reťazcami,
mapping si dorátame cez `indexOf` v normalizovanom kanonickom kontexte.

**Prečo `EpubBook` namiesto `String chapterText`:** kanonický text kapitoly (§4.5)
musí prísť z *epubx* parsera, nie z rendereru — inak `createHighlight` a
`findHighlight` operujú nad odlišnými textovými streammi a anchor sa systematicky
nenájde. Volajúci dodáva celú knihu + chapterId; `AnchorCodec` si interne
vytiahne kanonický text cez `CanonicalChapterText.extract(book, chapterId)`.

> **Investigačná závislosť:** sekcia 3.3 tabuľka musí potvrdiť (a) aký formát
> vracia `EpubView` selekcia, (b) **či sa selekcia objaví v kanonickom epubx
> texte kapitoly** na všetkých 4 fixtures. Bod (b) je make-or-break — ak zlyhá,
> M3 nemôže ani začať.

### Resolve algoritmus (`findHighlight`)

1. **Normalize** `chapterText`.
2. **Exact match** — hľadaj `anchor.quote` v okne ±200 znakov okolo
   `anchor.charOffset` (alebo celý text ak `charOffset` chýba).
3. Ak **jeden výskyt** → vráť ho.
4. Ak **viac výskytov** alebo žiadny v hint okne:
   - Skús `anchor.prefix + anchor.quote + anchor.suffix` (presný kontext)
   - Ak match → vráť pozíciu `quote` vo vnútri match-u
5. Ak **stále žiadny exact match**:
   - **Fuzzy substring search** cez sliding alignment (Smith-Waterman-style
     lokálne zarovnanie, NIE Levenshtein proti celému oknu).
   - **Algoritmus:** kĺzaj `needle` (`anchor.quote`) cez search okno ±500 znakov
     okolo `charOffset`. Pre každú kandidátnu pozíciu `i` v okne spočítaj edit
     distance medzi `needle` a substring `window[i : i+len(needle)+Δ]` kde
     `Δ ∈ [-10, +10]` (povolíme malý posun dĺžky na vloženia/zmazania).
   - **Metrika:** `similarity = 1 - editDistance / len(needle)` — menovateľ je
     dĺžka needle (nie okna), takže pre needle ~200 znakov a edit distance ~16
     dostaneme similarity ~0.92.
   - Threshold: `similarity ≥ 0.92` (cca 16 zmien v 200-znakovom citáte).
   - **Tie-break** keď viac matches má rovnakú similarity: vyhráva tá s najmenšou
     vzdialenosťou od `charOffset`. Ak `charOffset` chýba, vyhráva najskoršia
     pozícia v texte.
   - **Implementačná poznámka:** plný Smith-Waterman je O(n×m), čo pre okno
     1000 × needle 200 = 200k operácií — akceptovateľné. Ak by performance bola
     problém, fallback na sliding-window cez rolling hash filter pred edit
     distance (filter prejde do plnej kontroly len kandidátov s aspoň 80 % zhodou
     znakových bigramov).
6. Ak ani fuzzy nepomôže → vráť `null`.

> **Prečo nie pôvodný vzorec `1 - editDistance / max(len(needle), len(window))`:**
> matematicky nemohol fungovať. Aby needle ~200 znakov zarovnal celé okno 1000
> znakov, treba minimálne ~800 insertov → similarity ≤ 0.20, nikdy nie ≥ 0.92.
> Sliding alignment problém obíde — počíta sa proti najlepšiemu **lokálnemu**
> zarovnaniu, nie proti celému oknu.

Volajúca UI vrstva pri `null` zobrazí napr. „Táto kotva sa stratila po zmene
obsahu knihy" a používateľ môže anotáciu znova vytvoriť alebo zmazať.

### `createHighlight` implementácia (string-first, kanonický text)

```dart
static HighlightAnchor createHighlight({
  required EpubBook book,
  required String chapterId,
  required String rawSelectedText,
  required String rawPrefix,
  required String rawSuffix,
}) {
  // Kanonický text kapitoly — JEDINÝ zdroj pravdy (§4.5).
  // To isté použije aj findHighlight pri resolve.
  final canonicalChapter = CanonicalChapterText.extract(book, chapterId);

  final quote = normalize(rawSelectedText);
  final prefix = normalize(rawPrefix);
  final suffix = normalize(rawSuffix);

  // Cap prefix/suffix na 32 znakov (zoberieme posledných 32 z prefixu,
  // prvých 32 zo suffixu — najbližšie ku quote).
  final cappedPrefix = prefix.length > 32
      ? prefix.substring(prefix.length - 32)
      : prefix;
  final cappedSuffix = suffix.length > 32
      ? suffix.substring(0, 32)
      : suffix;

  // Cap quote (P3.1: long-selection — odložené robustnejšie riešenie).
  final cappedQuote = quote.length > 200 ? quote.substring(0, 200) : quote;

  // charOffset je len HINT — hľadáme v KANONICKOM texte (nie render-texte!).
  // Ak sa selekcia v kanonickom texte nenájde (napr. footnote ktorý epubx
  // nezahŕňa), charOffset ostane null a findHighlight to riešia fuzzy
  // matchom bez hint okna.
  final hintNeedle = cappedPrefix + cappedQuote + cappedSuffix;
  final hintMatch = canonicalChapter.indexOf(hintNeedle);
  final charOffset = hintMatch >= 0 ? hintMatch + cappedPrefix.length : null;

  return HighlightAnchor(
    chapterId: chapterId,
    quote: cappedQuote,
    prefix: cappedPrefix,
    suffix: cappedSuffix,
    charOffset: charOffset,
  );
}
```

### 5.5 Anchor vs. annotation record — oddelenie zodpovedností

`HighlightAnchor` (a `ReadingPositionAnchor`) sú **čisté lokátory** — odpovedajú na
otázku „**kde** v knihe". Neobsahujú žiadnu identitu, timestamp, ani metadata
o anotácii samotnej.

**Annotation record** (vznikne vo Fáze 3) musí mať tieto polia *navyše*:

```dart
class Annotation {
  final String id;            // CLIENT-GENERATED UUID v4 (stable across devices)
  final String bookId;        // FK na knihu
  final HighlightAnchor anchor;
  final String? note;         // používateľova poznámka (Fáza 3)
  final int? colorIndex;      // farba zvýraznenia (Fáza 3)
  final DateTime createdAt;   // ISO-8601 s milliseconds
  final DateTime updatedAt;   // pre LWW pri editácii poznámky
  final String deviceId;      // stable per-install ID (pre sync attribution)
  final bool isDeleted;       // soft-delete tombstone (pre sync — preferujeme nad hard delete)
}
```

**Prečo už teraz:** bez týchto polí Fáza 5 nedokáže:
- **Deduplicate** — dvaja klienti vytvoria highlight pred prvým sync-om;
  bez UUID by sa zlúčili do dvoch identických záznamov pri každom sync.
- **Merge edits** — používateľ upraví poznámku na PC, potom na mobile;
  bez `updatedAt` neviem ktorá verzia je novšia.
- **Replicate deletions** — bez tombstone (`isDeleted`) deletion na jednom
  zariadení sa nezreplikuje, alebo sa záznam „vzkriesi" pri ďalšom sync.

`HighlightAnchor` ale tieto polia **neobsahuje** — ostáva čistým lokátorom. Reusable
na čokoľvek čo potrebuje „bod v knihe": highlighty, bookmark-y, prípadne shared
links („pošli mi citát z tejto knihy odtiaľto").

> **Tento spec definuje len anchor.** Annotation schéma sa zafixuje v spec-u Fázy 3,
> ale ADR 0002 v dôsledkoch explicitne uvedie že schéma musí obsahovať vyššie
> uvedené polia. Tento „výhradzujem si miesto" prístup zabráni tomu aby Fáza 3
> implementácia začala bez týchto polí a potom vyžadovala migration.

## 6. Testovacia stratégia

**Unit testy nad reálnymi EPUB fixtures** v `test/fixtures/`:

### HighlightAnchor

1. **Round-trip:** zavolaj `createHighlight` s raw textom z `alice.epub` kapitoly
   (raw quote + raw prefix + raw suffix), encode JSON → decode → `findHighlight`
   v tej istej kapitole → vráti `AnchorRange` ktorý mapuje na pôvodný citát.

2. **Tolerance malých zmien:** vezmi text kapitoly, vlož 5 znakov pred quote,
   `findHighlight` cez fuzzy stále vie nájsť (s posunutou pozíciou).

3. **Disambiguácia:** quote ktorý sa v kapitole vyskytuje 2× (umelo:
   napr. spoločné slovo „the king said"), prefix + suffix správne určia
   ktorý výskyt.

4. **Strata kotvy:** quote vymazaný z textu → `findHighlight` vráti `null`, žiadny
   crash.

5. **Diakritika v `divina.epub`:** vytvor anchor nad reťazcom obsahujúcim
   talianske akcenty (`è`, `à`, `ò`), overí že NFC normalizácia funguje na encode
   aj na resolve.

6. **Normalizácia whitespace:** anchor vytvorený nad textom s mixed whitespace
   (newlines + spaces) sa resolve-ne aj keď chapter text má whitespace inak.

7. **Cap dlhej selekcie:** vytvorenie anchoru z 300-znakovej raw selekcie
   vyrobí `HighlightAnchor` s `quote.length == 200` (cap-nutý). Žiadny crash,
   warning v debugPrint.

### ReadingPositionAnchor

8. **Round-trip:** vytvor `ReadingPositionAnchor(chapterId: "...", charOffset: 5000,
   chapterLength: 12000)`, encode JSON → decode → polia rovnaké.

9. **Furthest-read merge:** mock dvoch eventov z rôznych zariadení
   (chapter 5 @ 100, chapter 3 @ 9000); pravidlo `(spine_index, charOffset)`
   lex order vráti správny event (chapter 5 wins, chapter 3 je menšia spine).

### CanonicalChapterText

10. **Determinizmus extraktora:** dva volania `CanonicalChapterText.extract`
    nad tou istou knihou + chapter id vrátia byte-identický string.

11. **Fixture coverage:** extract beží bez exceptionu na všetkých 4 fixtures
    (alice, pride, frankenstein, divina), výstup má rozumnú dĺžku (> 100 znakov).

### Sliding fuzzy

12. **Sliding fuzzy nájde insert:** vlož 5 znakov uprostred quote v kapitole,
    `findHighlight` vráti `AnchorRange` s `similarity ≥ 0.92`. (Tento test
    by s pôvodným zlým fuzzy vzorcom zlyhal — viď Resolve algoritmus §5.)

### Coverage

Cieľová coverage: ≥ 90 % v `lib/domain/anchor/`.

## 7. ADR 0002

Spec sa zafixuje v `docs/adr/0002-anchor-format.md`. ADR obsahuje:

- Kontext (rovnaký ako sekcia 1 tu, skrátený)
- Tabuľku z investigácie (sekcia 3.3)
- Rozhodnutie (oba anchor formáty + `chapterId` voľba)
- Dôvod (prečo dva osobitné, prečo href primárne, prečo fuzzy threshold 0.92)
- Dôsledky:
  - **Fáza 3 (anotácie) schéma** musí obsahovať polia z §5.5 (UUID, deviceId,
    createdAt, updatedAt, isDeleted tombstone) — aby Fáza 5 sync fungoval bez
    migrácie. `HighlightAnchor` sa serializuje ako JSON do stĺpca `anchor`.
  - **Fáza 5 (sync)** prenáša anchory ako súčasť append-only event log entries.
    Merge ordering pre reading position: `(spine_index, charOffset)` lexikograficky,
    *furthest read wins* ako default (viď §4.1).
  - **Reading position v M3** použije `ReadingPositionAnchor` namiesto súčasných
    `(chapterIndex, progressInChapter 0-1)` polí v `reading_positions` tabuľke.
    Po M3 môže byť potrebná **DB migrácia** `reading_positions` — chápeme to ako
    akceptovateľný cost; migrácia bude triviálna (drop dvoch stĺpcov, pridanie
    `anchor_json TEXT`).
  - **Cross-edícia limit:** anchor sa resolve-uje len v EPUB súbore s rovnakým
    content hashom (viď §4.3). Cross-edícia nie je cieľom MVP; book-identity ADR 0003
    túto hranicu zafixuje.
- Alternatívy zamietnuté (CFI via WebView — odmietnuté pre crash na Windows;
  paragraph index — odmietnuté pre nestabilitu pri reflow;
  render-height-based progress — odmietnuté pre device-dependence, viď §4.1 pozn.)
